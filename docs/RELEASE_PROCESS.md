## Introduction

The Zixflow React Native SDK uses a **two-stage automated deployment process**:

1. **Stage 1**: Semantic-release creates a git tag and GitHub release
2. **Stage 2**: The package is published to npm

This guide covers both the automated CI/CD process (recommended) and manual release procedures for fallback scenarios.

**Target Audience**: Maintainers, release engineers, and team members responsible for SDK releases.

## Version Management

### Semantic Versioning Strategy

The SDK follows [Semantic Versioning 2.0.0](https://semver.org/):

**Version Format**: `MAJOR.MINOR.PATCH`

- **MAJOR**: Incompatible API changes (breaking changes)
- **MINOR**: New functionality in a backwards-compatible manner
- **PATCH**: Backwards-compatible bug fixes

**Examples**:
- `1.0.0` → `2.0.0` - Breaking change (removed/changed API)
- `1.0.0` → `1.1.0` - New feature (added new method)
- `1.0.0` → `1.0.1` - Bug fix (fixed existing functionality)

**Pre-release versions**:
- `1.1.0-beta.1` - Beta release
- `1.1.0-alpha.1` - Alpha release

**Conventional Commits**:

The automated release process uses [Conventional Commits](https://www.conventionalcommits.org/) to determine version bumps:

- `feat:` → MINOR version bump
- `fix:` → PATCH version bump
- `BREAKING CHANGE:` in commit body → MAJOR version bump
- `chore:`, `docs:`, `refactor:` → No version bump

### Version Files

Version information is stored in multiple files and must be kept in sync:

1. **`package.json`**:
   ```json
   {
     "name": "zixflow-reactnative",
     "version": "1.0.0"
   }
   ```

2. **`android/src/main/res/values/customer_io_config.xml`**:
   ```xml
   <string name="customer_io_react_native_sdk_client_version">1.0.0</string>
   ```

3. **`CHANGELOG.md`** (auto-generated by semantic-release)

The automated release process updates all these files automatically.

- [ ] **CHANGELOG reviewed**: Ensure upcoming changelog entries are accurate (for manual releases)

## Release Process Overview

The Zixflow React Native SDK uses a **two-stage release process**:

```
┌─────────────────────────────────────────────────────────────┐
│                    STAGE 1: Git Tag                          │
│                                                              │
│  Push to main → semantic-release analyzes commits           │
│              → Updates version in files                      │
│              → Commits version changes                       │
│              → Creates git tag                               │
│              → Creates GitHub Release                        │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                   STAGE 2: npm Publish                       │
│                                                              │
│  Tag created → Checks out tagged commit                      │
│             → Builds package                                 │
│             → Publishes to npm (latest tag)                  │
│             → Triggers sample app builds                     │
└─────────────────────────────────────────────────────────────┘
```

## Automated CI/CD Release (Recommended)

The automated release process is triggered by **pushing to the `main` branch**.

### Stage 1: Git Tag Creation

**Workflow File**: `.github/workflows/deploy-sdk.yml`

**Job**: `deploy-git-tag`

**Process**:

1. **Trigger**: Push to `main` branch
   ```bash
   git push origin main
   ```

2. **semantic-release analyzes commits** since the last release
   - Uses Conventional Commits format
   - Determines next version (MAJOR, MINOR, or PATCH)

3. **Updates version files** using `scripts/update-version.sh`
   - `package.json`
   - `android/src/main/res/values/customer_io_config.xml`

4. **Commits changes** with message:
   ```
   chore: prepare for X.Y.Z [skip ci]
   ```

5. **Creates git tag**: `X.Y.Z`

6. **Pushes tag** to GitHub

7. **Creates GitHub Release** with auto-generated changelog

8. **Sends Slack notification** to `#mobile-deployments` channel

**Example Output**:

```
✓ semantic-release analyzed 5 commits
✓ Next version: 1.1.0
✓ Updated package.json
✓ Updated customer_io_config.xml
✓ Committed version changes
✓ Created tag: 1.1.0
✓ Pushed tag to GitHub
✓ Created GitHub Release
```

### Stage 2: npm Publishing

**Job**: `deploy-npm`

**Trigger**: Stage 1 completes successfully and a new release was published

**Process**:

1. **Checks out tagged commit** (e.g., `1.1.0`)

2. **Sets up Node.js 20** with npm registry

3. **Installs npm 11.5.1+** for OIDC authentication support

4. **Installs dependencies**:
   ```bash
   npm ci
   ```

5. **Runs pre-deploy script**:
   ```bash
   npm run pre-deploy
   ```
   This script:
   - Builds the TypeScript source
   - Generates type definitions
   - Prepares the package for publishing

6. **Publishes to npm** with `latest` tag:
   ```bash
   npm publish --tag latest
   ```

7. **Sends Slack notification** on success

8. **Triggers sample app builds** to verify the release

**Example Output**:

```
✓ Checked out commit for tag 1.1.0
✓ Installed dependencies
✓ Built package
✓ Published zixflow-reactnative@1.1.0 to npm
✓ Package available at https://www.npmjs.com/package/zixflow-reactnative
```

## Manual Release Process

Use the manual process if CI/CD is unavailable or for testing.

### Step 1: Update Version Files

Run the version update script:

```bash
./scripts/update-version.sh "1.1.0"
```

This updates:
- `package.json`
- `android/src/main/res/values/customer_io_config.xml`

Verify the changes:

```bash
git diff package.json
git diff android/src/main/res/values/customer_io_config.xml
```

### Step 2: Update CHANGELOG

Manually update `CHANGELOG.md` with release notes:

```markdown
## [1.1.0] - 2024-03-15

### Added
- New feature X
- Support for Y

### Fixed
- Bug in Z component

### Changed
- Improved performance of A
```

### Step 3: Commit Version Changes

Commit the version changes:

```bash
git add package.json android/src/main/res/values/customer_io_config.xml CHANGELOG.md
git commit -m "chore: prepare for 1.1.0 [skip ci]"
```

### Step 4: Create Git Tag

Create and push the git tag:

```bash
git tag -a 1.1.0 -m "Release 1.1.0"
git push origin main --tags
```

### Step 5: Build and Publish to npm

1. **Install dependencies**:
   ```bash
   npm ci
   ```

2. **Run pre-deploy script**:
   ```bash
   npm run pre-deploy
   ```

3. **Publish to npm**:
   ```bash
   npm publish --tag latest
   ```

### Step 6: Create GitHub Release

Create a GitHub Release manually:

1. Go to https://github.com/zixflow/zixflow-reactnative/releases/new
2. Select the tag `1.1.0`
3. Set release title: `1.1.0`
4. Copy release notes from CHANGELOG.md
5. Publish release

## Version Update Scripts

### `scripts/update-version.sh`

Updates version in `package.json` and `customer_io_config.xml`.

**Usage**:

```bash
./scripts/update-version.sh "1.1.0"
```

**What it does**:

1. Updates `package.json` using `npm version`
   ```bash
   npm version "1.1.0" --no-git-tag-version --allow-same-version
   ```

2. Updates `android/src/main/res/values/customer_io_config.xml` using `sd` CLI
   ```bash
   sd "<string name=\"customer_io_react_native_sdk_client_version\">.*</string>" \
      "<string name=\"customer_io_react_native_sdk_client_version\">1.1.0</string>" \
      android/src/main/res/values/customer_io_config.xml
   ```

3. Shows git diff to confirm changes

**Prerequisites**:
- `sd` CLI installed: `brew install sd` (macOS) or see [sd documentation](https://github.com/chmln/sd)

## Publishing to npm

### Automated Publishing (CI/CD)

Publishing happens automatically after Stage 1 (git tag creation) completes.

**Workflow**: `.github/workflows/deploy-sdk.yml` → `deploy-npm` job

### Manual Publishing

1. **Authenticate with npm**:
   ```bash
   npm login
   ```

2. **Verify package contents**:
   ```bash
   npm pack --dry-run
   ```

3. **Publish to npm**:
   ```bash
   npm publish --tag latest
   ```

4. **Verify publication**:
   ```bash
   npm view zixflow-reactnative
   ```

### npm Dist Tags

The SDK uses the following npm distribution tags:

| Tag | Description | Usage |
|-----|-------------|-------|
| `latest` | Stable release (production) | `npm install zixflow-reactnative` |
| `next` | Pre-release (beta testing) | `npm install zixflow-reactnative@next` |
| `beta` | Beta release | `npm install zixflow-reactnative@beta` |
| `alpha` | Alpha release | `npm install zixflow-reactnative@alpha` |

**Publishing with a specific tag**:

```bash
# Publish beta release
npm publish --tag beta

# Publish next release
npm publish --tag next
```

**Moving dist tag**:

```bash
# Promote beta to latest
npm dist-tag add zixflow-reactnative@1.1.0-beta.1 latest
```

## Post-Release Verification

After publishing a release, verify the deployment:

### 1. Verify npm Package

```bash
# Check package info
npm view zixflow-reactnative

# Check latest version
npm view zixflow-reactnative version

# Check dist-tags
npm view zixflow-reactnative dist-tags
```

### 2. Verify GitHub Release

- Visit: https://github.com/zixflow/zixflow-reactnative/releases
- Confirm the new release is published with correct:
  - Version number
  - Release notes
  - Assets (if any)

### 3. Verify Git Tag

```bash
git fetch --tags
git tag -l
git show 1.1.0
```

### 4. Test Installation

Create a test project and install the SDK:

```bash
npx react-native init TestApp
cd TestApp
npm install zixflow-reactnative
```

Verify installation:

```javascript
// Test import
import { Zixflow } from 'zixflow-reactnative';
console.log('Zixflow SDK loaded successfully');
```

### 5. Test Sample App

Build and run the example app with the new version:

```bash
cd example

# Install new version
npm install

# iOS
cd ios && pod install && cd ..
npx react-native run-ios

# Android
npx react-native run-android
```

### 6. Verify Native Dependencies

Check that native iOS and Android SDKs are correctly linked:

**iOS (CocoaPods)**:
```bash
cd example/ios
cat Podfile.lock | grep Zixflow
```

Expected output:
```
- Zixflow (1.0.0)
- ZixflowCommon (1.0.0)
- ZixflowDataPipelines (1.0.0)
```

**Android (Gradle)**:
```bash
cd example/android
./gradlew dependencies | grep zixflow
```

## Rollback Procedures

### npm Rollback

**Important**: You cannot delete a published version from npm. Instead, publish a new patch version with fixes.

1. **Deprecate the problematic version**:
   ```bash
   npm deprecate zixflow-reactnative@1.1.0 "This version has a critical bug. Please upgrade to 1.1.1"
   ```

2. **Publish a new patch version** with the fix:
   ```bash
   ./scripts/update-version.sh "1.1.1"
   git add . && git commit -m "fix: critical bug in 1.1.0"
   git push origin main
   ```

### Git Tag Rollback

**Warning**: Only delete tags that haven't been released to npm yet.

1. **Delete local tag**:
   ```bash
   git tag -d 1.1.0
   ```

2. **Delete remote tag**:
   ```bash
   git push origin :refs/tags/1.1.0
   ```

3. **Delete GitHub Release** (if created):
   - Go to https://github.com/zixflow/zixflow-reactnative/releases
   - Find the release
   - Click "Delete"

### Full Rollback (Pre-npm Publish)

If you catch an issue **before** npm publishing completes:

1. Delete the git tag (local and remote)
2. Delete the GitHub Release
3. Revert the version commit:
   ```bash
   git revert HEAD
   git push origin main
   ```

## Hotfix Process

For critical bugs requiring immediate release:

### 1. Create Hotfix Branch

```bash
git checkout main
git pull origin main
git checkout -b hotfix/1.0.1
```

### 2. Apply the Fix

```bash
# Make your changes
git add .
git commit -m "fix: critical issue with push notifications"
```

### 3. Update Version

```bash
./scripts/update-version.sh "1.0.1"
git add package.json android/src/main/res/values/customer_io_config.xml
git commit -m "chore: bump version to 1.0.1 [skip ci]"
```

### 4. Test the Fix

```bash
npm test
npm run lint

# Test on both platforms
cd example
npm install
npx react-native run-ios
npx react-native run-android
```

### 5. Merge to Main

```bash
git checkout main
git merge hotfix/1.0.1
git push origin main
```

### 6. Release Process

The automated CI/CD will handle the release, or follow the manual process.

### 7. Clean Up

```bash
git branch -d hotfix/1.0.1
```

## Native SDK Dependencies

The React Native SDK depends on native Zixflow SDKs for iOS and Android.

### iOS Native SDK

**Version**: 1.0.0

**Repository**: https://github.com/zixflow/zixflow-ios

**Modules used**:
- `ZixflowCommon`
- `ZixflowDataPipelines`
- `ZixflowMessagingPush`
- `ZixflowMessagingPushAPN`
- `ZixflowMessagingInApp`

**Integration**: CocoaPods or Swift Package Manager

**Podspec location**: `zixflow-reactnative.podspec`

### Android Native SDK

**Version**: 1.0.0

**Repository**: https://github.com/zixflow/zixflow-android

**Maven Group ID**: `com.zixflow.android`

**Modules used**:
- `zixflow-android:core`
- `zixflow-android:data-pipelines`
- `zixflow-android:messaging-push`
- `zixflow-android:messaging-push-fcm`
- `zixflow-android:messaging-in-app`

**Integration**: Gradle/Maven Central

### Updating Native SDK Versions

When a new native SDK version is released:

1. **Update iOS Podspec** (`zixflow-reactnative.podspec`):
   ```ruby
   s.dependency "Zixflow", "~> 1.1"
   ```

2. **Update Android Gradle** (`android/build.gradle`):
   ```gradle
   implementation "com.zixflow.android:core:1.1.0"
   ```

3. **Test compatibility** with the new native SDK version

4. **Update package.json metadata** (if needed):
   ```json
   {
     "zixflowNativeiOSSdkVersion": "~> 1.1",
     "zixflowNativeAndroidSdkVersion": "1.1.0"
   }
   ```

5. **Document breaking changes** in CHANGELOG.md

6. **Release new React Native SDK version**

## Known Issues & Workarounds

### Issue 1: npm Publish Authentication Failures

**Symptom**: `npm publish` fails with authentication error

**Cause**: npm authentication token expired

**Solution**:
1. Log in again: `npm login`
2. Verify: `npm whoami`
3. Retry: `npm publish`

---

### Issue 2: GitHub App Token Expiration (CI/CD)

**Symptom**: CI/CD workflow fails at git push step with "Authentication failed"

**Cause**: GitHub App credentials expired or misconfigured

**Solution**:
1. Verify GitHub App credentials in repository secrets:
   - `CIO_APP_CLIENT_ID`
   - `CIO_APP_SECRET`
2. Regenerate credentials if needed
3. Update repository secrets

---

### Issue 3: semantic-release No Release Published

**Symptom**: semantic-release runs but doesn't create a new release

**Cause**: No commits since last release that trigger version bump

**Solution**:
- Ensure commits follow Conventional Commits format:
  - `feat:` for new features (MINOR bump)
  - `fix:` for bug fixes (PATCH bump)
  - `BREAKING CHANGE:` in commit body (MAJOR bump)
- Check commit messages: `git log --oneline`

---

### Issue 4: Pod Install Fails After SDK Update

**Symptom**: `pod install` fails with dependency conflicts after updating native SDK

**Cause**: CocoaPods cache or conflicting versions

**Solution**:
```bash
cd example/ios
rm -rf Pods Podfile.lock
pod repo update
pod install
```

---

### Issue 5: Type Definitions Not Generated

**Symptom**: TypeScript types missing after build

**Cause**: Build script didn't run or failed silently

**Solution**:
```bash
# Clean build
rm -rf lib/
npm run prepare
npm run build
```

Check that `lib/` directory contains `.d.ts` files.

## Appendix

### Workflow File Reference

**File**: `.github/workflows/deploy-sdk.yml`

**Key sections**:

1. **Trigger**:
   ```yaml
   on:
     push:
       branches: [main]
   ```

2. **Stage 1 - Git Tag**:
   ```yaml
   jobs:
     deploy-git-tag:
       runs-on: ubuntu-latest
       steps:
         - uses: actions/checkout@v4
         - uses: cycjimmy/semantic-release-action@v4
   ```

3. **Stage 2 - npm Publish**:
   ```yaml
   deploy-npm:
     needs: [deploy-git-tag]
     if: needs.deploy-git-tag.outputs.new_release_published == 'true'
     steps:
       - run: npm ci
       - run: npm run pre-deploy
       - run: npm publish --tag latest
   ```

### semantic-release Configuration

**File**: `.releaserc.json`

**Configuration**:
```json
{
  "tagFormat": "${version}",
  "branches": ["main", "beta", "alpha"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    "@semantic-release/changelog",
    ["@semantic-release/exec", {
      "verifyReleaseCmd": "./scripts/update-version.sh ${nextRelease.version}"
    }],
    ["@semantic-release/git", {
      "assets": [
        "CHANGELOG.md",
        "package.json",
        "android/src/main/res/values/customer_io_config.xml"
      ]
    }],
    "@semantic-release/github"
  ]
}
```
